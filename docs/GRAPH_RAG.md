# 图谱 RAG、引用与安全护栏

`POST /qa` 现在采用轻量的 Graph-RAG 流程：先由规则识别问题类型，再以 Cypher 从 Neo4j 检索结构化事实，最后基于检索结果组织回答。它不依赖外部模型服务，因此在本地环境也可运行；后续接入 LLM 时，应只把 `retrieval` 上下文作为生成输入，并保持原有引用与安全检查不变。

## 响应契约

除既有的 `answer`、`intent` 和 `data` 外，问答接口还返回：

- `retrieval`：检索策略、是否有可追溯证据、使用的上下文记录数。
- `citations`：每项含稳定 ID、来源标题、图谱字段或 CSV 定位信息、支撑的事实和来源类型。
- `safety`：动作（`allowed`、`review_required` 或 `blocked`）、风险等级、触发关键词、提示及固定免责声明。

## 安全策略

- 包含胸痛、呼吸困难、昏迷、严重过敏、药物过量等紧急信号的请求会被 `blocked`，不会访问知识图谱或给出医疗处置建议。
- 包含剂量、加减量、停换药、处方、妊娠/哺乳、儿童或肝肾功能不全等词的请求为 `review_required`。系统可以展示知识检索结果，但必须由医生或药师结合完整病史确认。
- 无论风险等级，系统始终声明其为知识学习和临床辅助工具，不能替代专业人员或急救服务。

## 引用边界

治疗方案和药品适应症优先引用 Neo4j 中的指南来源字段。药物相互作用、禁忌等当前演示数据没有独立的文献字段，因此引用会明确定位到相应的本地 CSV 数据集；这表示可追溯的数据来源，而不表示已完成外部临床证据验证。

指南来源必须在 `data/source_registry.csv` 中登记。登记表包含来源 ID、发布方、版本、发布日期、链接、获取日期和审核状态；`scripts/clean.py` 会拒绝导入未登记的指南来源。当前演示数据均标记为 `unverified_demo`，在医学审核完成前不得视为已验证临床证据。
